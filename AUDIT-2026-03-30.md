# Necronomichat v2 — Full Token Efficiency Audit
**Date:** 2026-03-30
**Scope:** System prompt, agent loop, context folding/compaction, tool schemas, model routing, MCP tools

---

## Executive Summary

The $10/response problem has multiple contributing causes, not one smoking gun. The codebase is a Frankenstein hybrid of the old hermes-agent verbose loop (~1,650 lines inline in run_agent.py) and the new nchat compact loop (~675 lines in nchat/loop.py). Compact mode is well-designed but has gaps where old-mode systems still fire or new-mode systems silently drop features.

**The biggest token sinks, in order of impact:**

| # | Issue | Est. Token Waste/Call | Fix Difficulty |
|---|-------|----------------------|----------------|
| 1 | MCP tools (Obsidian+Playwright+Blender) sent every call | ~6,000 tokens | Medium |

## TIER 1: Critical (Causing the $10 responses)

### 1.1 — MCP Tools Registered Unconditionally (~6,000 tokens/call)

**Where:** `model_tools.py:172-177` → `discover_mcp_tools()`

All configured MCP servers dump their entire tool schemas into every API call. Your config has:
- Obsidian MCP: ~15 tools
- Playwright MCP: ~22 tools
- Blender MCP: ~25 tools
- **= 62 MCP tools × ~100 tokens each = ~6,200 tokens**

These are sent on EVERY turn, whether you're asking about 3D modeling or checking the weather. For a 15-turn conversation, that's ~93K tokens just from MCP schemas.

**Fix options:**
- **Quick:** Add `check_fn` to MCP tools that checks if the server is running/reachable
- **Better:** Lazy loading — register a single `mcp_discover` tool; model calls it to load a specific server's tools when needed
- **Best:** Intent-based loading via UTILITY crystal classifying the first message

### 1.2 — Fold + Compaction Double-Fire in Code Medium

**Where:** `nchat/loop.py:879-909`

In `_run_code_cast`, every turn does:
1. Step 10 (line 883): `fold_code_context()` → calls auxiliary LLM to generate fold summary
2. Immediately after (line 898): `compressor.should_compress()` → calls auxiliary LLM AGAIN

Both can fire on the same turn. Two auxiliary LLM calls covering overlapping message ranges. The compaction system doesn't know about fold nodes and may re-summarize the fold summary itself.

**Fix:** Add early return after successful fold:
```python
if did_fold:
    fold_count += 1
    continue  # skip compaction — folding already handled it
```

### 1.3 — Reasoning Tokens Compound Across Turns (Hidden Cost)

**Where:** `run_agent.py:2702-2705` and `run_agent.py:6762-6763`

For thinking models (Claude extended thinking, Moonshot, etc.), the `reasoning_content` field is stored in messages and re-sent on every subsequent turn. Turn N carries ALL reasoning from turns 1 through N-1. This grows quadratically.

A single turn of extended thinking can be 5K-20K tokens. After 10 turns, you're re-sending 50K-200K tokens of reasoning content that the model already produced — and this is INVISIBLE in the message history because it's a hidden field.

**Fix:** Strip `reasoning_content` from messages older than the last 2-3 turns before building the API payload. The model doesn't need its old chain-of-thought to continue working.

### 1.4 — Iteration-Limit Summary Sends Full Conversation (Up to 4x)

**Where:** `run_agent.py:6276-6463`

When max iterations are hit, `_handle_max_iterations` does:
1. Builds `api_messages` from the **full message list** (could be 100K+ tokens)
2. Sends entire thing to auxiliary model for summary
3. If that fails → sends entire thing to PRIMARY model (Opus)
4. If that fails → retries BOTH again

That's up to **4 full-context API calls** just for the summary. If the conversation is already 100K tokens, the summary generation alone costs 400K+ input tokens.

**Fix:** Truncate the conversation before sending to the summarizer. The fold system's approach (400 chars/message) would reduce a 100K conversation to ~5K for summarization.

## TIER 2: High Impact (Steady Token Drain)

### 2.1 — Tool Result Cap at 100K Characters (~25K Tokens)

**Where:** `run_agent.py:5884` — `MAX_TOOL_RESULT_CHARS = 100_000`

A single `read_file` on a minified JS file or a `terminal` running `pip install` can dump 25K tokens into context. The old compactor's `_prune_old_tool_results` protects the last 60 messages from pruning, so these giant results persist.

**Fix:** Reduce to 30K chars (~7,500 tokens). Also add a character cap to `read_file` (currently has line cap of 2000 but no char cap — a single line of minified code can be megabytes).

### 2.2 — Terminal Output Cap at 50K Characters

**Where:** `tools/terminal_tool.py:1149` — `MAX_OUTPUT_CHARS = 50000`

Similar issue. Most useful terminal output is under 5K chars. A `pip install` or `cargo build` can fill this easily.

**Fix:** Reduce to 20K-25K chars.

### 2.3 — Compaction Serializer Too Generous

**Where:** `agent/context_compressor.py:215-222`

The compaction serializer keeps 3,000 chars per tool result and 3,000 chars per assistant message. For a 40-message middle section, that's up to 120K chars (~30K tokens) fed to the auxiliary model just for summarization.

Compare: the fold system caps at 400 chars per tool result and 600 chars per code block.

**Fix:** Align compaction serialization limits with fold limits. 500-800 chars per message is plenty for a summarizer.

### 2.4 — Compaction Summary Output Allows 24K Tokens

**Where:** `agent/context_compressor.py:349`

The summarizer is allowed to generate `max_tokens = summary_budget * 2 = up to 24,000 tokens`. For a tool that's supposed to REDUCE context, this is backwards. A 24K summary replacing 30K of messages is barely a reduction.

**Fix:** Hard-cap summary output at 4K-6K tokens.

### 2.5 — Previous Summary Accumulates in Iterative Compaction

**Where:** `agent/context_compressor.py:266-270`

Each compaction pass feeds the previous summary back as input. If the previous summary was 8K tokens and new turns are 20K, the summarizer receives 28K. This grows monotonically.

**Fix:** Truncate the previous summary to 2K chars before feeding it to the next compaction pass.

## TIER 3: Moderate Impact (Cleanup & Correctness)

### 3.1 — Honcho Turn Context Silently Dropped (BUG)

**Where:** Old loop injects at `run_agent.py:6752`. New loop's `_build_loop_api_messages()` (lines 2689-2746) has NO reference to `_honcho_turn_context`.

If Honcho is configured, later-turn memory recall is silently lost in compact mode. Not a token waste issue — it's a correctness bug that means Honcho integration is broken in compact mode.

**Fix:** Add `_honcho_turn_context` injection to `_build_loop_api_messages()`, or pass it through the bridge.

### 3.2 — Conversation-Mode Folding is Dead Code

**Where:** `nchat/loop.py:572-577`, `nchat/fold.py` lines 382-481

The comment at loop.py:572 explicitly disables conversation folding. All the `fold_conversation_context()` code and `_CONVERSATION_FOLD_PROMPT_TEMPLATE` in fold.py are dead code. Only `fold_code_context()` runs (in code medium path).

This means conversation mode relies entirely on the old reactive compaction, which is the more expensive and less effective system.

**Decision needed:** Either activate conversation folding with better heuristics, or remove the dead code. The current state is confusing.

### 3.3 — SOUL.md Few-Shot Examples (~800 Token Savings)

**Where:** `~/.hermes/SOUL.md` (loaded by `nchat/call.py:52-68`)

7 user/agent exchange examples consuming ~1,200 tokens. For Opus/Sonnet, 2-3 examples suffice (~400 tokens). The prose identity description already calibrates tone.

**Fix:** Trim to 2-3 of the most representative examples.

### 3.4 — Skills Preamble in Compact Mode (~80 Tokens)

**Where:** `agent/prompt_builder.py:408-422`

`build_skills_system_prompt()` includes behavioral guidance ("scan skills before replying", "offer to save as skill after difficult tasks") even in compact mode. This contradicts compact mode's design principle.

**Fix:** Pass `compact=True` and skip the preamble in compact mode.

### 3.5 — Active Gates List Redundant with Tools Parameter (~200 Tokens)

**Where:** `nchat/call.py:127-131`

The compact system prompt includes a list of tool names + one-liners. The model already receives this information via the API's `tools` parameter. Redundant.

**Fix:** Remove the gate list from the system prompt. The tools parameter is sufficient.

## TIER 4: Dead Code & Frankenstein Wiring (Maintenance Debt)

### 4.1 — Dead Code in Old Loop's Post-Section

**Where:** `run_agent.py:8262-8332`

Lines 8262-8292 (loom recording) and 8319-8332 (compact circle review) are in the old verbose loop's post-section. Compact mode returns at line 6680 and never reaches these. They're dead code.

### 4.2 — `_total_tool_calls_this_turn` Never Set

**Where:** `run_agent.py:8323`

`getattr(self, '_total_tool_calls_this_turn', 0)` — this attribute is never assigned anywhere. Even if the code were reachable, `tool_call_count` would always be 0.

### 4.3 — Auth-Refresh Logic Missing from Compact Mode

**Where:** Old loop has Codex/Nous/Anthropic 401 retry logic at lines 7230-7249. The new `_make_loop_api_call()` does NOT have this. If auth tokens expire mid-conversation in compact mode, the call fails without retry.

### 4.4 — Codex Ack Continuation Missing from Compact Mode

**Where:** Old loop handles Codex intermediate acknowledgment messages at lines 8105-8131 (model outputs a planning message without tool calls → auto-continue). `run_cast()` has no equivalent. Codex models in compact mode would terminate prematurely on ack messages.

### 4.5 — Sequential Tool Execution Duplicates Dispatch Logic

**Where:** `_execute_tool_calls_sequential()` (lines 5919-6200) has inline tool dispatch duplicating `_invoke_tool()` (lines 5666-5728). Any new tool must be added to BOTH places.

### 4.6 — `_handle_max_iterations()` Unreachable in Compact Mode

**Where:** `run_agent.py:6276-6463` — ~190 lines only reachable from the old verbose loop. Could be removed if verbose mode is eventually deprecated.

### 4.7 — Verbose Honcho Block Injects CLI Documentation (~300 Tokens)

**Where:** `run_agent.py:3217-3228`

The verbose system prompt includes a full CLI command reference (`hermes honcho status`, `hermes honcho mode [hybrid|honcho]`, etc.). The model doesn't need to know CLI syntax to function.

## Recommended Fix Order

1. **MCP lazy loading** (biggest single win, ~6K tokens/call)
2. **Skip compaction after successful fold** (prevents double LLM calls in code medium)
3. **Strip old reasoning_content from messages** (prevents quadratic growth)
4. **Truncate conversation before iteration-limit summary** (prevents 4x full-context calls)
5. **Reduce tool result caps** (100K→30K chars, terminal 50K→20K)
6. **Align compaction serializer with fold limits** (3K→500-800 chars/msg)
7. **Cap compaction summary output** (24K→4-6K tokens)
8. **Fix Honcho turn context bug** (correctness, not cost)
9. **Trim SOUL.md examples** (easy win, ~800 tokens)
10. **Remove dead code** (lines 8262-8332, compact budget warning branch, etc.)
